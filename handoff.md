# 24-H 自动行驶小车工程交接说明

> 最后核对：2026-07-22
> 仓库：<https://github.com/zkj070612-gif/24-H>
> 本文基于提交 `e475c3a` 后的代码整理，目标是让下一位开发者能直接编译、烧录、看懂主流程并继续实车调参。

## 1. 项目现状

本工程是 2024 年自动行驶小车 H 题的 MSPM0G3507 版本，硬件为 Yahboom YB-CVT01-V1.0 小车。当前比赛主线已经具备：

- 上电初始化 OLED、MPU6050、编码器、电机、定时器和板载 Flash。
- 使用 OLED 和按键选择 1～4 号任务模式。
- 使用八路循迹模块的 UART 数字量输出，而不是八个独立 GPIO。
- 直线路段使用 MPU6050 航向角保持，弯道路段使用八路循迹 PID。
- 使用白/黑线稳定时间、最短行驶时间和转弯角度共同判断状态切换。
- 八路模块超过 500 ms 没有有效数据时刹车并进入错误状态。
- 使用蜂鸣器和 RGB 灯提示启动、过点、完成和故障。
- OLED 实时显示模式、圈数、状态、航向角及八路传感器值。

已知实车进度：固件曾烧录并能驱动车轮动作；由于当时手边没有完整赛道，四种模式的整圈路线和参数尚未完成实地验收。

当前代码**没有**接入 K230、HC-06 蓝牙、NRF24L01、无线图传、电磁铁、MOS 驱动或 UBEC。这些属于后续扩展，不要把它们当成当前 24-H 主线的一部分。

## 2. 开发与烧录环境

- MCU：MSPM0G3507SPMR（Cortex-M0+）。
- 开发工具：Keil MDK，现有工程按 ARM Compiler 6 配置；原开发环境为 MDK-ARM 6.22。
- TI SDK：MSPM0 SDK `2.10.00.04`。
- 原 SDK 路径：`C:\ti\mspm0_sdk_2_10_00_04`。
- Keil 工程：`keil\YB_CVT01_H8.uvprojx`。
- 输出目录：`OBJ\`。
- 目标 HEX：`OBJ\YB_CVT01_H8.hex`。
- 烧录工具：TI UniFlash。

仓库不提交 `OBJ/`、`*.hex`、`*.axf` 等编译产物，所以刚克隆后看不到 HEX 是正常现象，必须先在 Keil 中 Build/Rebuild。

若 SDK 安装路径不同，需要修改 Keil 工程的 Include Paths，以及工程中指向 SDK 的链接/库路径。不要把个人的 `*.uvoptx` 提交到仓库。

## 3. 上电后的完整流程

入口在 `main.c`：

1. `bsp_init()` 初始化底层硬件。
2. 立即刹车，关闭蜂鸣器和 RGB。
3. `switch_mode()` 显示模式选择界面：KEY1 循环切换 1～4，KEY2 确认。
4. `WaitForTrackingModule()` 等待八路循迹模块稳定：
   - 全局上电计时不足 20 秒时显示 `IR warming`；
   - 随后显示 `Waiting IR UART`；
   - 每秒通过 UART0 发送一次 `$0,0,1#`，请求模块连续输出数字量；
   - 只有收到第一帧完整有效数据后才允许启动。
5. `Competition_Init(mode)` 根据模式建立初始航向目标并进入第一个直线状态。
6. 主循环持续运行协作式调度器、比赛状态机和 OLED 刷新。

注意：如果 OLED 一直停在 `Waiting IR UART`，说明程序尚未收到合法的八路数字帧。优先检查模块供电、TX/RX 是否交叉、波特率以及模块是否真的输出 `$D,...#` 格式，不是电机程序卡死。

## 4. 四种比赛模式

| 模式 | 当前代码中的路线含义 | 完成条件 |
| --- | --- | --- |
| 1 | A→B | 第一个直线终点检测到后停车 |
| 2 | A→B→C→D→A | 完成一圈后停车 |
| 3 | A→C→B→D→A | 完成一圈后停车 |
| 4 | 按模式 3 的路线连续运行 | 完成四圈后停车 |

程序只使用前进差速和刹车。当前比赛状态机不安排倒车或原地反转。

OLED 的 `S` 状态编号与 `CompetitionState` 对应：

| S 值 | 状态 |
| ---: | --- |
| 0 | IDLE，空闲 |
| 1 | STRAIGHT_FIRST，第一段直线 |
| 2 | ARC_FIRST，第一段循迹弯道 |
| 3 | STRAIGHT_SECOND，第二段直线 |
| 4 | ARC_SECOND，第二段循迹弯道 |
| 5 | FINISHED，任务完成 |
| 6 | ERROR，传感器超时或任务总超时 |

## 5. 代码模块划分

### 5.1 入口与任务状态机

| 文件 | 作用 | 当前主线状态 |
| --- | --- | --- |
| `main.c` | 上电流程、循迹模块握手、主循环和 OLED 状态显示 | 核心启用 |
| `BSP/control/control.c/.h` | 模式选择、路线状态机、直线航向控制、弯道切换、超时保护、声光提示 | 核心启用 |
| `BSP/task/task.c/.h` | 基于 `Get_Time()` 的协作式周期调度器 | 核心启用 |

调度任务如下：

| 周期 | 函数 | 用途 |
| ---: | --- | --- |
| 5 ms | `Get_EulerAngles()` | 从 MPU DMP 获取欧拉角并滤波 |
| 10 ms | `Get_CalibratedAngles()` | 生成 0～360° 的 `calibratedYaw` |
| 20 ms | `encoder_update()` | 汇总编码器增量和里程计数 |
| 20 ms | `Motion_Handle()` | 更新轮速测量；只有 `g_start_ctrl` 置位时才执行其速度环输出 |

调度器不抢占，也不补跑错过的周期。不要在主循环或控制状态中加入长时间 `delay_ms()`，否则 MPU、编码器、UART 前台解析和控制刷新都会被拖慢。

### 5.2 八路循迹与 UART0

| 文件 | 作用 | 当前主线状态 |
| --- | --- | --- |
| `BSP/IR/bsp_ir.c/.h` | UART0 接收中断、帧缓存、完整性解析、最后有效帧时间 | 核心启用 |
| `APP/app_irtracking_eight.c/.h` | 八路数据复制、黑线判断、加权误差和循迹 PID | 核心启用 |
| `BSP/usart0.c/.h` | UART0 发送函数；故意丢弃 `printf` 输出 | 核心启用 |

UART0 参数及接线：

| 信号 | MCU | 连接 |
| --- | --- | --- |
| UART0 TX | PA10 | 接八路模块 RX |
| UART0 RX | PA11 | 接八路模块 TX |
| 波特率 | 115200 bit/s | 8 位数据，使用 SysConfig 当前设置 |
| GND | 共地 | MCU 与模块必须共地 |

接收帧要求：

- 一帧必须从 `$` 开始，以 `#` 结束。
- 帧类型必须为 `$D...#`。
- 必须同时找到 `x1`～`x8` 八个字段，每个值只能是 `0` 或 `1`。
- 当前约定：`1 = 白底`，`0 = 黑线`。
- 80 字节接收缓存只保留一帧；前台未取走完整帧时，新字节会被丢弃。
- `IR_LastFrameAgeMs()` 超过 500 ms，比赛状态机立即刹车并进入 ERROR。

UART0 已被循迹模块独占。`fputc()` 当前只返回字符而不真正发送，目的是防止遗留 `printf` 文本破坏模块协议。要调试请优先使用 OLED/Keil Debug；若要加日志串口，另配 UART，不要直接恢复 UART0 的 `printf`。

循迹算法当前参数：

- 权重：`{-20, -15, -10, -5, 5, 10, 15, 20}`。
- 基础速度：`IRR_SPEED = 70`。
- PID：`Kp=-28.0`、`Ki=0`、`Kd=-2.8`。
- 转向输出限幅：±45。
- 如果八路全为白色，沿用上次误差方向，以 ±20 继续寻线；从未有偏差时误差为 0。

### 5.3 路线状态判断与航向控制

`BSP/control/control.c` 中的重要参数：

| 参数 | 当前值 | 含义 |
| --- | ---: | --- |
| `CONTROL_PERIOD_MS` | 10 ms | 状态机控制周期 |
| `LINE_STABLE_MS` | 40 ms | 直线终点黑线稳定时间 |
| `ARC_EXIT_STABLE_MS` | 60 ms | 弯道出口白底稳定时间 |
| `STRAIGHT_MIN_MS` | 400 ms | 直线段最短持续时间 |
| `ARC_MIN_MS` | 700 ms | 弯道段最短持续时间 |
| `ARC_MIN_YAW_DEG` | 135° | 弯道退出所需最小航向变化 |
| `STRAIGHT_SPEED_CMD` | 53 | 直线基础指令 |
| `HEADING_KP` / `KD` | 1.20 / 0.80 | 直线航向 PD |
| `HEADING_TURN_LIMIT` | 28 | 航向修正限幅 |
| `MIN_WHEEL_CMD` / `MAX_WHEEL_CMD` | 20 / 90 | 直线左右轮指令范围 |
| 非 4 模式总超时 | 90 s | 超时进入 ERROR |
| 模式 4 总超时 | 240 s | 超时进入 ERROR |

直线终点不是“看到一次黑线”就切换：代码要求此前已经稳定看到白底，然后黑线稳定 40 ms，并且本直线至少行驶 400 ms。

弯道出口同时要求：全白稳定 60 ms、进入弯道至少 700 ms、相对弯道起点的航向变化至少 135°。这样可以减少刚入弯时误判出口。

两组对角线修正集中在同一个文件，按圈索引：

```c
static const float s_first_diagonal_deg[4]  = {-35.0f, -31.5f, -29.0f, -30.3f};
static const float s_second_diagonal_deg[4] = { 37.4f,  38.3f,  37.0f,  37.3f};
```

如果实车第一次 A→C 转向完全相反，先同时反转两组数组的正负号，再做小幅调参。不要一开始同时修改循迹 PID、航向 PID和对角线角度，否则无法判断是哪一层导致偏差。

### 5.4 电机、编码器与 PID

| 文件 | 作用 | 当前主线状态 |
| --- | --- | --- |
| `BSP/Motor/bsp_motor.c/.h` | TIMA0 四路 PWM、左右 H 桥方向、刹车、死区和输出限幅 | 核心启用 |
| `BSP/Motor/bsp_encoder.c/.h` | 两个 AB 相编码器中断、20 ms 增量、总计数和里程累计 | 启用 |
| `APP/app_motor.c/.h` | 轮速换算、运动抽象和可选速度闭环 | 部分启用 |
| `APP/bsp_PID_motor.c/.h` | 电机 PID、航向 PID及多套遗留 PID | 核心调用但含遗留代码 |

硬件/几何常量：

- 左电机为 Motor1，右电机为 Motor2。
- 编码器一圈脉冲常量：`1170.0`。
- 轮周长：`204.204 mm`。
- 车体 APB 常量：`157.09 mm`。
- PWM 软件限幅：`MAX_SPEED = 850`。
- 电机死区补偿：`MOTOR_DEAD_ZONE = 75`。

重要技术债：比赛主线调用 `Set_PID_Motor(left, right, 0)`。该函数名看起来像常规编码器速度闭环，但当前实现把传入的左右指令送入 `PID_Location_Calc()`，而 PID 的 `target_val` 通常仍为 0；它并没有在这条调用链中直接使用编码器实测速度。另一套真正使用 `motor_data.speed_mm_s[]` 的速度闭环存在于 `Motion_Set_Speed()` / `Motion_Handle()`，但比赛状态机没有走这条路径。

因此，接手人调参时应把当前 `Set_PID_Motor()` 视为“带 PID 形式映射的电机指令输出”，不要未经实测就假定它是标准轮速闭环。若要重构成闭环，先单独验证左右编码器符号、单位、目标速度和电机方向，再替换比赛主线，避免整车行为突然反向。

### 5.5 MPU6050 与姿态

| 文件/目录 | 作用 | 当前主线状态 |
| --- | --- | --- |
| `BSP/mpu6050/bsp_mpu6050.*` | 软件 I²C 读写 MPU6050 | 核心启用 |
| `BSP/mpu6050/get_mpu6050.*` | DMP 数据、低通滤波、航向角转换 | 核心启用 |
| `BSP/eMPL/` | InvenSense DMP 支持代码 | 核心启用 |
| `BSP/filter/`、`BSP/kalman/` | 滤波工具及遗留算法 | 部分使用 |

板载 IMU 插座使用：

- SDA：PA0。
- SCL：PA1。
- MPU6050 地址：0x68（AD0 接地）。

`calibratedYaw` 被转换到 0～360°。小车上电和模式选择期间应保持静止；直线控制和弯道退出条件都依赖航向角，IMU 松动、方向改变或车体震动会直接影响路线。

### 5.6 显示、按键和声光提示

| 模块 | 引脚/作用 |
| --- | --- |
| OLED 软件 I²C | SCL=PA15，SDA=PA30 |
| KEY1 | PA18，模式加一 |
| KEY2 | PB23，确认模式 |
| KEY3 | PB20，当前比赛主线未使用 |
| 蜂鸣器 | PB5 |
| WS2812/RGB | PB10 |

声光含义：

- 黄灯 + 150 ms 蜂鸣：任务启动。
- 青灯 + 100 ms 蜂鸣：通过状态切换点。
- 绿灯 + 300 ms 蜂鸣：任务完成，绿灯保持。
- 红灯 + 500 ms 蜂鸣：错误停车，红灯保持。

### 5.7 已初始化但没有参与路线决策的模块

- `BSP/ultrasonic/`：当前只启动了提供毫秒时基的定时器，路线状态机没有读取超声波距离。
- `BSP/flash/`：`bsp_init()` 会调用 `W25Q64_Init()` 检查板载 Flash，但任务逻辑没有保存或读取比赛数据。
- `APP/app_irremote.*` 与 `BSP/bsp_ir_receiver.*`：红外遥控代码仍被编译，但 `IRRMOTE` 在 `BSP/config.h` 中为 0，主循环也不调用 `IR_Control_Car()`。
- SysConfig 中的 ADC 和旧红外板相关引脚：当前 UART 八路比赛主线不使用。
- `veer_pid`、`k230_pid` 等多套 PID：属于原工程遗留/预留代码，当前比赛状态机没有使用 K230 通信或控制。

特别注意：`W25Q64_Init()` 会一直循环，直到读到 W25Q64 或 W25Q32 的 ID。如果程序连模式选择界面都进不去，除 MPU 初始化外，也要检查板载 Flash/SPI；缺失或读 ID 异常可能导致启动永久停在这里。

## 7. 编译和 UniFlash 烧录

1. 安装 MSPM0 SDK 2.10.00.04 和带 ARM Compiler 6 的 Keil MDK。
2. 用 Keil 打开 `keil\YB_CVT01_H8.uvprojx`。
3. 检查 SDK Include Paths 是否指向本机安装位置。
4. 执行 Rebuild，确认 0 Error。
5. 确认生成 `OBJ\YB_CVT01_H8.hex`。
6. 打开 UniFlash，选择 MSPM0G3507 和实际使用的调试器。
7. 在 Program 页面选择该 Intel HEX，执行 Load Image/Program。
8. 第一次烧录或刚改过电机代码时应架空驱动轮，避免小车突然冲出桌面。
9. 烧录完成后断开调试占用，将小车放到起点，静止上电并选择模式。

## 8. 建议的接车验收顺序

不要直接从模式 4 开始跑。建议按以下顺序排查：

1. **静态上电**：能进入模式选择；OLED、按键、蜂鸣器和 RGB 正常。
2. **八路通信**：白底时 OLED 显示 `11111111`；用黑胶带依次压传感器时对应位变 0；不应长时间停在 `Waiting IR UART`。
3. **电机方向**：架空轮子跑模式 1，确认左右轮都向前；紧急情况能刹车。
4. **IMU 方向**：缓慢转动车体，确认 OLED Yaw 连续变化，跨 0/360° 时不出现错误的大跳变。
5. **制作最小测试道**：白色哑光底板 + 约 1.8 cm 黑胶带，先校准八路模块黑白阈值。
6. **模式 1**：只验证直线航向保持和终点黑线停车。
7. **单个弯道**：验证循迹左右方向、出弯全白判定和 135° 航向门槛。
8. **模式 2/3**：再调整两组对角线修正角。
9. **模式 4**：最后逐圈调整数组的四个元素并验证总超时。

每次只改一类参数并记录：先传感器黑白与机械安装，再电机方向，再循迹 PID，再直线航向 PD，最后对角线修正角。

## 9. 已知风险和待办

- 完整赛道尚未实测，模式 2～4 的路线、圈数和终点状态需要逐项验证。
- 两组对角线角度来自原 Yahboom H 题测试值，对轮胎、重心、电池电压、场地摩擦和 IMU 安装非常敏感。
- 循迹“任意一路为 0 就算黑线”，交叉线、污点或宽线可能触发直线状态切换，需要在真实赛道观察。
- 弯道丢线时会按上次误差方向继续寻线，没有独立的最大丢线时间；总体仍受 UART 500 ms 超时和任务总超时保护。
- `W25Q64_Init()` 无退出超时，Flash 故障会让整机无法进入模式界面。
- 源码中保留较多原厂示例和乱码注释；改功能前应以实际调用链为准，不要仅凭函数名判断是否启用。
- 当前仓库没有编译产物；交接时如果需要可另行提供当次验收通过的 HEX 及对应 Git 提交号，但不要把无法追溯版本的 HEX 当作唯一源码。

## 10. 后续硬件扩展边界

目前讨论/采购过的 K230、HC-06、5 V 电磁铁、MOS 驱动、5 V UBEC 和续流二极管都**未进入本工程的软件和 SysConfig**。

继续扩展时至少要做到：

- HC-06 或 K230 若走串口，新增独立 UART 和协议解析；UART0 必须继续留给八路循迹模块。
- K230 自带 Wi-Fi 只说明它能做无线网络通信，不等于 MSPM0 已经能收到其数据；两块板之间仍需定义 UART/SPI 等本地接口和消息协议。
- 电磁铁必须由独立 5 V 大电流电源供电，经逻辑电平 MOS 驱动，并在线圈两端并联续流二极管；不能由 MSPM0 GPIO 或板载 5 V 小电流口直接带动。
- 新增 MOS 控制前先在原理图和 SysConfig 中选择无冲突 GPIO，再增加上电默认关闭、任务超时关闭和故障关闭逻辑。
- 无线图传只有题目明确要求远程图像或 K230 端画面必须传回时才准备；它不能替代小车控制链路。

这些扩展建议单独建分支，不要在 24-H 基线尚未完成赛道验收时直接混入主状态机。
